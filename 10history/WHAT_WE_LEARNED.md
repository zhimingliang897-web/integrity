# What We Learned - 照片整理项目经验总结

## 1. 文件的修改时间 (mtime) 不可信

从手机、网盘、聊天工具导出的照片，mtime 几乎一定是**导出/下载的时间**，而不是拍摄时间。

在我们的测试数据中，所有文件的 mtime 都是 `2025-02-05`（批量复制的那天），完全无法反映真实拍照时间。

**教训**: 永远不要把 mtime 当作拍摄日期的可靠来源。如果 EXIF 和文件名都提取不到日期，应该诚实地标记为"未知"，而不是用一个错误的日期来冒充。

## 2. 文件名本身是最被低估的元数据来源

实际数据中 **91%** 的文件日期（65/71）是从文件名解析出来的，而不是 EXIF。

常见的文件名模式：
- `2018-02-11 184617.jpg` — 日期时间直接写在文件名里
- `1464689880000.jpeg` — Unix 毫秒时间戳
- `1681371254840_1737332047006_77.jpeg` — 复合时间戳（第一个是拍摄时间，第二个可能是传输时间）

**教训**: 先从文件名提取日期是纯字符串操作，比打开文件读 EXIF 快得多。在处理几万张文件时，这个差异非常显著。

## 3. 去重不需要对每个文件都算完整哈希

三级过滤策略让大部分文件跳过了昂贵的完整哈希计算：

```
72 个文件
  → 第一级（文件大小）: 68 个直接通过，只有 4 个需要进一步检查
  → 第二级（前 4KB 哈希）: 又排除了 2 个
  → 第三级（完整 MD5）: 只有 2 个文件需要计算完整哈希
```

**教训**: 文件大小本身就是一个很好的"免费"去重指标。大部分文件的大小是唯一的，根本不需要读取文件内容。

## 4. EXIF 读取的内存陷阱

`Pillow` 的 `Image.open()` 虽然是惰性加载，但在读取 EXIF 时仍然会解析图片头部并创建 Image 对象。处理几万张文件时，如果不及时关闭文件句柄，内存会持续增长。

`exifread` 库是更好的选择——它只读取文件头部的 EXIF 数据（通常几 KB），完全不涉及图片解码。

**教训**: 如果只需要元数据，不要用图片处理库。用专门的元数据读取库。

## 5. PNG 文件几乎没有拍摄时间信息

PNG 格式设计上就不像 JPEG 那样内嵌 EXIF。虽然 PNG 有 tEXt 块可以存储文本元数据，但实际上：
- 截图工具生成的 PNG 极少写入拍摄时间
- 手机截图的 PNG 文件名通常是随机字符串或序号
- 只有少数现代应用会在 PNG 的 eXIf 块中写入 EXIF

**教训**: 对 PNG 文件，文件名是唯一靠得住的日期来源。如果文件名也无法解析，就应该归入"未知日期"。

## 6. 重复文件应该保留证据

最初设计是直接跳过重复文件，只在日志里记一行。但用户（合理地）想知道"你到底去掉了什么"。

最终方案：
- 重复文件单独复制到 `_重复文件/` 文件夹
- 生成 `_重复对照清单.txt`，记录每个重复文件对应的原件路径

**教训**: 去重是一个不可逆操作（即使只是复制时的去重）。给用户保留审计线索，比沉默地跳过要好得多。

## 7. Dry-run 模式是必需品，不是可选功能

在处理几万张个人照片时，一旦出错就很难恢复。`--dry-run` 让用户可以完整预览所有操作，确认无误后再实际执行。

在开发过程中，dry-run 也帮助我们快速验证日期提取和去重逻辑的正确性。

## 8. 待解决的问题

- **进度持久化**: 几万张文件的复制过程可能很长，如果中途中断需要从头重跑。未来可以加一个 `.progress.json` 文件记录已处理的文件哈希，断点续传。
- **Windows 终端编码**: 中文在某些 Windows 终端中显示为乱码，tqdm 的进度条在非 UTF-8 终端中也可能有显示问题。
